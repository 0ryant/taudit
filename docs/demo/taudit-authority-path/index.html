Scenario A · authority propagation (developer-tool path)

Hidden authority, drawn.

workflows/before.yml is an ordinary two-job pipeline: a build job produces an artifact, a deploy job ships it through a third-party action. Two things make it dangerous and neither is visible reading the YAML top-to-bottom: authority is granted workflow-wide (contents/packages/id-token: write applies to every step), and the executable is mutable (vendor/deploy-action@v2 is a moving tag). Whoever moves that tag inherits write authority, an OIDC identity, and three production secrets.

before.yml · hidden authority
# taudit graph --view authority before.yml
step: Third-party production deploy [untrusted]
  has_access_to → CLOUD_DEPLOY_TOKEN
  has_access_to → PROD_API_KEY
  has_access_to → RELEASE_TOKEN
  has_access_to → GITHUB_TOKEN
  uses_image    → vendor/deploy-action@v2 [unpinned]

# taudit scan before.yml
22 findings · 11 critical
7 untrusted_with_authority (exploitable core)
after.yml · reduced authority
# least privilege · SHA-pinned · scoped secrets
step: Third-party production deploy [pinned]
  default permissions: contents: read
  action pinned → full commit SHA
  secrets ← protected production env
  GITHUB_TOKEN split → (build) / (deploy)

# taudit diff before.yml after.yml
findings 22 → 8 (−14)
7 untrusted_with_authority → 0
4 unpinned actions → 0 · workflow-wide write removed

What changed: after.yml applies the standard remediation with no new tooling — default permissions: contents: read with per-job opt-in, every action pinned to a full-length commit SHA, and deploy secrets scoped to the protected production environment. taudit diff (results/diff.txt) states it precisely: 22 → 8, all 7 untrusted_with_authority criticals and all 4 unpinned-action findings removed. Graphs: before-authority.png · after-authority.png.

What remains · honest residual (after.yml is not zero, by design)

2 critical — GITHUB_TOKEN propagation. Actions injects this token into every job; its reach is structural. This is the floor, not a regression.

3 high — the three secrets still reach a third-party action. It is now pinned and reviewed, so this is residual third-party trust, not an open exploit. Removing it means moving to OIDC or a first-party deploy path.

1 high — id-token: write on deploy could be scoped tighter still.

1 low / 1 info — PROD_API_KEY is a static credential (use OIDC), plus an OIDC-without-attestation note.

Baselined authority — the gate. taudit baseline init after.yml records those 8 residual findings as the accepted floor (.taudit/baselines/). From then on:

$ taudit baseline diff after.yml
after.yml: 0 NEW, 0 FIXED, 8 PRE-EXISTING

0 NEW is the point. A future PR that re-broadens a permission or unpins an action shows up as NEW authority movement and fails the gate — while the known floor stays quiet. Captured in baseline/baseline-diff.txt.

Scenario B · exploit path (PATH-helper hijack)

A multi-hop kill-chain, traced end to end.

Scenario A is authority propagation — broad authority reaching an untrusted sink. Scenario B is sharper: a concrete, multi-hop exploit path that taudit graph --view exploit traces end to end. exploit-before.yml extends $GITHUB_PATH with a writable directory before a deploy action runs, poisoning the helper the action resolves while the Cloudflare deploy authority is in scope.

exploit-before-killchain · taudit graph --view exploit
PATH mutation            echo "$HOME/.local/bin" >> $GITHUB_PATH
   ↳ PATH-selected helper   npx / wrangler resolved via poisoned PATH
   ↳ deploy action          cloudflare/wrangler-action@v3 [unpinned]
   ↳ authority artifact     deploy secret payload
        CLOUDFLARE_API_TOKEN / SECRET_ALPHA exposed

# taudit graph --view exploit · path_count
exploit-before: 1 path   exploit-after: 0 paths
21 → 3
total findings · before → after
8 → 0
critical · before → after
1 → 0
--view exploit paths

exploit-after.yml removes the chain by deleting the PATH mutation (so helper resolution can't be redirected), pinning every action to a SHA, and dropping the unused id-token: write. The 3 residuals are high (Cloudflare secrets reaching the now-pinned third-party action) and one medium (structural GITHUB_TOKEN) — no criticals, no exploit path. exploit-after-killchain.dot is empty (path_count 0 — proof of removal). Reproduce it

No hand-edited outputs.

Requires the taudit binary (cargo build -p taudit --release) and Node 18+ for image rendering. reproduce.ps1 regenerates both scenarios end to end — and asserts the vendored _assets/site.css still matches the algol.cc source by sha256.

1
taudit scan
JSON + SARIF findings for before/after of both scenarios → findings/. 22, 8, 21, 3.
2
taudit map · graph
Authority matrix per step → authority-matrix/; DOT/Mermaid → SVG → PNG into graph/ (no system graphviz needed).
3
taudit diff
before → after diff, terminal + JSON → results/. Plus scan receipts → receipts/.
4
taudit baseline
baseline init records the 8-finding floor; baseline diff proves 0 NEW → baseline/.
reproduce.ps1 (excerpt)
P=docs/demo/taudit-authority-path

taudit scan  $P/workflows/before.yml --format json  -o $P/findings/before.findings.json
taudit map   $P/workflows/before.yml --format text > $P/authority-matrix/before.map.txt
taudit graph $P/workflows/before.yml --view authority --format dot > $P/graph/before-authority.dot
taudit diff  $P/workflows/before.yml $P/workflows/after.yml --format terminal > $P/results/diff.txt
taudit baseline init $P/workflows/after.yml --root $P --captured-by taudit-authority-path-demo
taudit baseline diff $P/workflows/after.yml --root $P
→ after.yml: 0 NEW, 0 FIXED, 8 PRE-EXISTING

# Scenario B — exploit view (kill-chain). path_count 1 → 0.
taudit graph $P/workflows/exploit-before.yml --view exploit --format dot > $P/graph/exploit-before-killchain.dot
