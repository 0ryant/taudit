# Visual Studio Marketplace Extension Contract

Status: implemented v1 product contract for the live `taudit` VS Code
extension published through Visual Studio Marketplace under publisher `algol`.

Live listing:
<https://marketplace.visualstudio.com/items?itemName=Algol.taudit-vscode>

Doctrine basis:

- `engineering-doctrine/doctrine/principles/build.md`: build surfaces must be
  explicit; tooling is replaceable, contracts are not.
- `engineering-doctrine/doctrine/principles/merge-path-evidence-and-pipeline-integrity.md`:
  binding gates must fail clearly and leave retrievable evidence.
- `engineering-doctrine/doctrine/principles/semantic-versioning.md`: public
  commands, settings, defaults, and result shapes are versioned contract.

This document defines the product contract the extension implementation,
Marketplace manifest, README, tests, and release notes must continue to
satisfy.

## Contract Identity

| Field | Value |
|-------|-------|
| Publishable unit | VS Code extension package (`.vsix`) |
| Marketplace publisher | `algol` |
| Proposed extension id | `algol.taudit-vscode` |
| Contract id | `dev.taudit.vscode-extension.v1` |
| Contract version | `1.0.0` |
| Primary manifest | extension root `package.json` |
| Execution model | typed VS Code adapter over the local `taudit` CLI |
| Default mode | `verify` for workspace policy checks; `scan` remains advisory |
| Default workspace target | `.github/workflows/` |
| Out of scope | policy authoring, arbitrary shell wrapper, automatic CI changes, automatic SARIF upload, remote attestation, tsign/axiom enforcement |

The extension follows SemVer independently from the `taudit` CLI. Every
extension release MUST state the default or minimum supported `taudit` CLI
version and MUST NOT publish from CLI metadata alone; the Marketplace product
has its own manifest, version, tests, screenshots, README, and release notes.

## Product Model

The extension is an editor-facing adapter for the same operator surface exposed
by `taudit scan`, `taudit verify`, and `taudit graph`.

It MUST expose typed commands and settings rather than raw argument passthrough.
It MUST keep these controls distinct:

- policy invariants used by `verify`
- ignore file used before finding evaluation
- suppressions used to waive or tag specific findings
- baselines used to distinguish new findings from accepted pre-existing ones

The extension is not a generic YAML linter. taudit models CI/CD authority
propagation as a deterministic graph; findings, SARIF, editor diagnostics,
graph diagrams, and verify gates are views over that graph.

## Commands

The extension MUST contribute these commands in v1:

| Command | CLI mapping | Contract |
|---------|-------------|----------|
| `taudit.verifyWorkspace` | `taudit verify [PATH...] --policy <path>` | Runs the configured policy gate over configured workflow roots. Preserves exit `0`/`1`/`2` as pass, violations, and config error in the editor result surface. |
| `taudit.scanWorkspace` | `taudit scan [PATH...]` | Runs advisory scan over configured workflow roots and opens findings output. It MUST NOT be treated as a merge gate inside the extension. |
| `taudit.scanFile` | `taudit scan <active-file>` | Runs advisory scan for the active supported pipeline file. It MUST fail clearly when the active editor is not a supported CI/CD file. |
| `taudit.graphAuthority` | `taudit graph [PATH...] --view authority` | Produces the authority graph for configured workflow roots or the active file/selection. |
| `taudit.graphExploit` | `taudit graph [PATH...] --view exploit` | Produces the exploit-candidate graph for configured workflow roots or the active file/selection. |
| `taudit.showOutput` | none | Focuses the taudit output channel and the most recent result artifact generated by the extension. |

The extension MAY add narrower convenience commands in minor releases when they
map to typed taudit features. It MUST NOT add `extraArgs`, `command`, `shell`,
`script`, or arbitrary CLI passthrough in v1.

## Settings Surface

Settings are the public configuration contract. They MUST be declared in the
extension manifest and validated before invoking `taudit`.

| Setting | Type | Default | Applies to | CLI mapping | Contract |
|---------|------|---------|------------|-------------|----------|
| `taudit.binaryPath` | string | `taudit` on `PATH` | all | executable path | Must resolve to an executable `taudit` binary before command execution. Missing binary is a config error, not a silent no-op. |
| `taudit.platform` | enum: `auto`, `github-actions`, `azure-devops`, `gitlab`, `bitbucket` | `auto` | all | `--platform` | Must match CLI enum tokens supported by the bundled/minimum CLI. |
| `taudit.workflowPaths` | array of workspace-relative strings | `[ ".github/workflows/" ]` | all | positional path args | Empty entries ignored; each value becomes one argv element. |
| `taudit.verify.policyPath` | workspace-relative file or directory | `.taudit/policy/` | `verify` | `--policy` | Required for `taudit.verifyWorkspace`. It is not a config directory, ignore file, suppressions file, or baseline root. |
| `taudit.verify.includeBuiltin` | boolean | `false` | `verify` | `--include-builtin` | Built-ins are off in `verify` unless explicitly enabled. |
| `taudit.verify.ignorePartial` | boolean | `false` | `verify` | `--ignore-partial` | Must be visible in result summaries when enabled. |
| `taudit.verify.format` | enum: `text`, `json`, `sarif` | `json` | `verify` | `--format` | Verify format is mode-specific and separate from scan/graph output choices. |
| `taudit.scan.format` | enum: `terminal`, `json`, `sarif`, `cloudevents` | `json` | `scan` | `--format` | Scan format is advisory/output-only; it does not turn scan into a gate. |
| `taudit.controls.ignoreFile` | workspace-relative file | auto-discover | `scan`, `verify` | `--ignore-file` | If configured, the path must exist. Otherwise CLI auto-discovery applies. |
| `taudit.controls.suppressionsFile` | workspace-relative file | auto-discover | `scan`, `verify` | `--suppressions` | If configured, the path must exist. Otherwise CLI auto-discovers `.taudit-suppressions.yml` and `.taudit/suppressions.yml`. |
| `taudit.controls.suppressionMode` | enum: `downgrade`, `tag-only` | `downgrade` | `scan`, `verify` | `--suppression-mode` | `downgrade` can affect severity gating; `tag-only` is metadata-only for `verify`. |
| `taudit.controls.baselineRoot` | workspace-relative directory | workspace root | `scan`, `verify` | `--baseline-root` where supported | Root containing `.taudit/baselines/`. |
| `taudit.graph.format` | enum: `json`, `dot`, `mermaid`, `summary` | `mermaid` | graph commands | `--format` | DOT and Mermaid may render in editor previews; JSON remains the canonical graph artifact. |
| `taudit.maxHops` | positive integer | CLI default | all | `--max-hops` | Reject non-integer, zero, negative, or excessive values before invoking CLI. |
| `taudit.severityThreshold` | string | empty | `scan`, `verify` | `--severity-threshold` | Empty means no threshold override; otherwise the value must be one of `critical`, `high`, `medium`, `low`, or `info`. |
| `taudit.runOnSave` | boolean | `false` | editor automation | command invocation | Must be opt-in. Current scaffold runs `scanFile` on save for supported pipeline files. |

Settings validation MUST happen before process invocation. Invalid enum values,
invalid booleans, invalid integers, missing explicit paths, and missing policy
for `verify` MUST fail clearly.

## Command Mapping

For a normalized setting object, the extension MUST build deterministic argv
arrays and invoke `taudit` with a process API that does not use a shell string.

Example settings:

```json
{
  "taudit.workflowPaths": [".github/workflows/", ".github/actions/"],
  "taudit.verify.policyPath": ".taudit/policy/",
  "taudit.verify.includeBuiltin": true,
  "taudit.verify.format": "json",
  "taudit.controls.suppressionsFile": ".taudit-suppressions.yml",
  "taudit.controls.suppressionMode": "tag-only",
  "taudit.controls.baselineRoot": "."
}
```

`taudit.verifyWorkspace` MUST invoke the equivalent argv:

```text
taudit verify
  --policy .taudit/policy/
  --platform auto
  --include-builtin
  --suppressions .taudit-suppressions.yml
  --suppression-mode tag-only
  --baseline-root .
  --format json
  --no-color
  .github/workflows/
  .github/actions/
```

The extension MUST NOT display this as a copy-paste shell command when any
argument is user-controlled or secret-adjacent. It may display a redacted,
structured argument list for debugging.

## Controls

### Policy

`taudit.verify.policyPath` is required for `verify`. If it is missing,
unreadable, or resolves outside the workspace without an explicitly documented
exception, `taudit.verifyWorkspace` MUST fail before invoking the CLI.

The extension MUST describe policy as the invariant bundle used for gate
decisions. It MUST NOT overload policy to mean suppressions, ignore rules,
baseline state, or general configuration.

### Ignore File

If `taudit.controls.ignoreFile` is set, the path must exist and must be passed
as `--ignore-file`. If unset, the extension should report whether CLI
auto-discovery found `.tauditignore` when that information is available.

Ignore rules apply before gate evaluation and are separate from suppressions.

### Suppressions

If `taudit.controls.suppressionsFile` is set, the path must exist and must be
passed as `--suppressions`. If unset, CLI auto-discovery may load
`.taudit-suppressions.yml` or `.taudit/suppressions.yml`.

The extension MUST surface the active `suppressionMode`. `tag-only` must not be
presented as making `verify` pass by itself. Critical waiver behavior remains
delegated to taudit; critical suppressions require bounded justification and
expired waivers do not apply.

### Baselines

`taudit.controls.baselineRoot` points to the root containing
`.taudit/baselines/`. Baselines are opt-in and per-pipeline. The extension MUST
surface whether baseline state is found, missing, unused, or unknown.

For `verify`, baseline behavior MUST preserve taudit semantics:

- new findings can fail the gate
- pre-existing non-critical findings may be accepted by baseline
- critical findings still count toward exit `1` unless explicitly waived
- a future typed `gateOnAll` setting, if added, must override baseline-new-only
  behavior explicitly rather than by hidden default change

### ADO Enrichment

ADO enrichment is deferred from the current scaffold. If it is added later, it
must require org, project, and PAT source together. The PAT value must be
retrieved from VS Code SecretStorage or an equivalent secure secret source at
execution time, passed to the child process without logging, and never stored
in workspace settings, output channels, artifacts, telemetry, or graph
metadata.

## Result Surfaces

The extension MUST expose results in editor-native surfaces without changing
taudit semantics.

| Surface | Contract |
|---------|----------|
| Command result | Reports command, target paths, active controls, taudit version, exit code, and outcome. |
| Output channel | Shows stdout/stderr with secrets redacted and without raw environment dumps. |
| Problems / diagnostics | MAY project findings from JSON or SARIF into VS Code diagnostics. Diagnostic severity MUST follow taudit severity and suppression metadata. |
| Artifacts | Stores selected JSON, SARIF, DOT, Mermaid, or summary output in extension storage, or a future explicit artifact-directory setting if one is added. |
| Graph preview | MAY render DOT/Mermaid and MUST link back to the exact graph artifact used for rendering. |
| Status bar / notification | MAY show latest outcome, but MUST NOT hide exit `2` behind a generic failure. |

The result summary SHOULD include the same control facts as the GitHub action
summary:

```text
taudit mode: verify
taudit version: <version>
policy: .taudit/policy/
include built-ins: true|false
ignore file: <path|not found|not used>
suppressions: <path|not found|not used> (mode=<downgrade|tag-only>)
baseline root: <path> (<found|missing|unused|unknown>)
partial graph policy: normal|ignore-partial
gate: <all findings|new findings + unwaived critical pre-existing|advisory>
exit: <0|1|2> (<pass|violations|config-error>)
```

The extension MUST preserve taudit exit semantics:

| Exit | Extension outcome | Meaning |
|------|-------------------|---------|
| `0` | pass | No policy violation or advisory command completed. |
| `1` | violations | Findings/violations crossed the active gate. |
| `2` | config-error | Configuration, usage, parse, missing policy, missing binary, or cannot-decide error. |

`scan` remains advisory/bootstrap in v1. The extension MUST NOT treat `scan` as
equivalent to `verify` unless a future typed setting deliberately adds and
tests explicit scan-gating semantics.

## Manifest And Packaging Constraints

The extension manifest MUST include, at minimum:

- `name`
- `displayName`
- `publisher: algol`
- `version`
- `engines.vscode`
- `description`
- `categories`
- `license`
- extension entrypoint fields
- command contributions
- configuration schema for every public setting above

Packaging and publishing MUST use `@vscode/vsce` or a documented compatible
tool. Marketplace publishing requires a Visual Studio Marketplace publisher and
an Azure DevOps PAT with Marketplace `Manage` scope. Automated publish SHOULD
use `VSCE_PAT` and MUST NOT broaden PAT scope without a documented exception.
Local operator publish paths MUST inject that PAT with `tsafe` or an equivalent
declared-scope secret wrapper; they MUST NOT pass PATs on command lines or rely
on ad hoc shell exports. Hosted CI publish lanes may use a CI secret variable.

Marketplace assets and README/CHANGELOG images MUST satisfy VS Marketplace
rules: no user-provided SVG icon and no non-HTTPS or untrusted SVG images.

The first publish MUST NOT happen until a packaged VSIX has been installed and
smoke-tested on a supported VS Code client.

## Security Invariants

The extension MUST fail readiness if any invariant below is broken:

- No raw CLI passthrough in v1.
- No shell command construction from user inputs.
- Explicit path settings are normalized and constrained to the workspace unless
  a deliberate exception is documented and tested.
- `verify` without policy fails before invoking `taudit`.
- Missing explicit `policyPath`, `ignoreFile`, `suppressionsFile`, or
  `baselineRoot` paths fail clearly.
- `verify` does not include built-ins unless `includeBuiltin` is true.
- `ignorePartial` is explicit and visible; partial/unknown graph coverage is
  not silently hidden.
- ADO PAT values are stored only in the editor secret store and are masked or
  omitted from every log, summary, artifact, and diagnostic.
- The extension does not request repository write access or modify CI files in
  v1.
- Result rendering never treats exit `2` as pass.

## Compatibility Rules

- Patch: wrapper fixes, docs, packaging metadata, result rendering fixes, and
  compatible output additions.
- Minor: new typed settings, new optional commands, new result views, or
  additional supported output formats/platforms that do not change defaults.
- Major: removing/renaming commands or settings, changing defaults, introducing
  raw passthrough, changing exit behavior, changing verify/scan/graph mapping,
  or changing where artifacts are written by default.

The extension MUST state the minimum supported `taudit` CLI version. If a
setting maps to a CLI flag unavailable in older versions, the extension MUST
fail clearly or hide/disable that setting for the selected binary version.

## Required Contract Tests

- Manifest declares every command and setting in this contract.
- Unknown, invalid, or partial settings fail before invoking `taudit`.
- `verifyWorkspace` without `policyPath` fails before invoking `taudit`.
- `verifyWorkspace` excludes built-ins by default and includes them only with
  `includeBuiltin: true`.
- Each setting maps to exactly one intended argv fragment or resolver behavior.
- Unset optional settings emit no argv.
- Path injection attempts are passed as data, not shell syntax.
- Missing explicit policy, ignore, suppressions, or baseline paths fail clearly.
- Auto-discovered `.tauditignore`, `.taudit-suppressions.yml`,
  `.taudit/suppressions.yml`, and `.taudit/baselines/` state is visible in
  result summaries where available.
- `suppressionMode: tag-only` does not make `verify` pass by itself.
- Baseline behavior covers new findings, pre-existing non-critical findings,
  unwaived criticals, waived criticals, and any future explicit `gateOnAll`
  control.
- `ignorePartial` is represented in summaries and diagnostics.
- `graphAuthority` and `graphExploit` both produce graph artifacts.
- If ADO enrichment is added later, it is all-or-none and PAT values remain
  absent from settings JSON, argv displays, output channels, diagnostics,
  summaries, and artifacts.
- VSIX packaging contents exclude development-only files and include required
  Marketplace metadata.
- A packaged VSIX install activates and runs each contributed command against a
  fixture workspace.

## Non-Goals For v1

- No automatic policy authoring or remediation editing.
- No automatic CI workflow mutation.
- No built-in GitHub Code Scanning or Azure DevOps upload.
- No third-party storage, SIEM upload, tsign attestation, or axiom decision
  publishing.
- No web extension support unless the runtime boundary is separately designed.
- No remote-workspace support unless binary execution, artifact storage, and
  trust boundaries are explicitly tested.
- No replacement for platform linters such as actionlint.
- No direct Marketplace publish before a real extension artifact, VSIX package,
  and install smoke exist.
