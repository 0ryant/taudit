# Release trust

This document describes what a tagged `taudit` release ships and how to verify it.

## What ships with each `vX.Y.Z` tag

Every version tag produces:

- GitHub Release archives for Linux x86_64, Linux arm64, macOS x86_64, macOS arm64, and Windows x86_64
- one `.sha256` checksum file per release archive
- one SPDX 2.3 dependency SBOM: `taudit-vX.Y.Z.spdx.json`
- one CycloneDX 1.5 dependency SBOM: `taudit-vX.Y.Z.cyclonedx.json`
- a crates.io publish of the `taudit` crate after the tagged quality gate and archive jobs succeed

## Release gate

The tag workflow reruns the same release-critical checks before publishing:

- `cargo fmt --all -- --check`
- `cargo clippy --workspace --all-targets -- -D warnings`
- `cargo test --workspace`
- `cargo deny check licenses bans sources`
- `cargo audit`

A separate `security` workflow (`.github/workflows/security.yml`)
re-runs `cargo deny check advisories bans licenses sources` on every
PR plus a weekly Monday cron, and runs `taudit scan` against
`.github/workflows/` as a hard gate (`taudit-scans-taudit`).

## Verifying a release archive

Download an archive and its matching checksum from the GitHub Release page, then verify it locally.

### Unix

```bash
curl -LO https://github.com/0ryant/taudit/releases/download/<tag>/taudit-x86_64-linux.tar.gz
curl -LO https://github.com/0ryant/taudit/releases/download/<tag>/taudit-x86_64-linux.tar.gz.sha256
sha256sum -c taudit-x86_64-linux.tar.gz.sha256
```

### macOS

```bash
shasum -a 256 -c taudit-aarch64-macos.tar.gz.sha256
```

### Windows PowerShell

```powershell
$expected = (Get-Content .\taudit-x86_64-windows.zip.sha256).Split(' ')[0]
$actual = (Get-FileHash .\taudit-x86_64-windows.zip -Algorithm SHA256).Hash.ToLower()
if ($expected -ne $actual) { throw 'checksum mismatch' }
```

## Verifying the SBOMs

Download whichever SBOM your tooling consumes from the release page:

```bash
# SPDX 2.3
curl -LO https://github.com/0ryant/taudit/releases/download/<tag>/taudit-<tag>.spdx.json
jq '.packages[].name' taudit-<tag>.spdx.json

# CycloneDX 1.5
curl -LO https://github.com/0ryant/taudit/releases/download/<tag>/taudit-<tag>.cyclonedx.json
jq '.components[].name' taudit-<tag>.cyclonedx.json
```

Both SBOMs cover Rust workspace dependencies resolved from
`Cargo.lock` at release time. SPDX is generated by `cargo-sbom`;
CycloneDX is generated by `cargo-cyclonedx`. Tools like
Dependency-Track, Trivy, and Grype prefer CycloneDX; SPDX is preferred
by SBOM-aware compliance tooling (FOSSology, OSS Review Toolkit).

## Trust boundary

The release workflow is defined in `.github/workflows/release.yml` and
publishes only after the tag build passes the release quality gate and
all archive jobs complete.

This release trust surface currently includes:

- SHA-256 checksums for every release archive
- two SBOM formats (SPDX 2.3 + CycloneDX 1.5)
- a publish-after-build dependency chain in the release workflow
  (the `publish` job needs `create-release`, `sbom`, and `build`)

It does **not yet** include cryptographically signed release artifacts
— that is tracked below as future work.

## Future work — signed releases (minisign)

The intent is to sign every release archive with
[minisign](https://jedisct1.github.io/minisign/) (Ed25519, no PKI,
single-file public key) so consumers can verify provenance with one
command and a single committed public key.

### Why minisign vs cosign

- **No keyless / no Sigstore dependency.** Consumers verify offline
  with the public key committed to this repo.
- **One static binary** (`minisign`) on every major OS — no Docker, no
  Fulcio, no Rekor lookups.
- **Smaller blast radius.** A single Ed25519 keypair, rotated
  manually, is easier to reason about than a Sigstore identity tied
  to a CI runner's OIDC token.

cosign is the right call for projects that need transparency-log
provenance; minisign is the right call for a binary CLI distributed
through GitHub Releases.

### Signing recipe (when implemented)

1. **Maintainer (offline, one time).** Generate the keypair on a
   trusted laptop:

   ```bash
   minisign -G -p taudit.pub -s ~/.taudit-minisign.key
   ```

   Commit `taudit.pub` to `release/taudit.pub` in this repo. Keep
   `~/.taudit-minisign.key` offline; do **not** ship it to CI.

2. **Maintainer (each release, locally after a tag is pushed).** After
   the release workflow has uploaded the archives, download each
   archive, sign it, and upload the signature:

   ```bash
   for asset in taudit-*.tar.gz taudit-*.zip; do
     gh release download "$TAG" -p "$asset"
     minisign -S -s ~/.taudit-minisign.key -m "$asset"
     gh release upload "$TAG" "$asset.minisig" --clobber
   done
   ```

3. **Consumer (verification).**

   ```bash
   curl -LO https://raw.githubusercontent.com/0ryant/taudit/main/release/taudit.pub
   curl -LO https://github.com/0ryant/taudit/releases/download/<tag>/taudit-x86_64-linux.tar.gz
   curl -LO https://github.com/0ryant/taudit/releases/download/<tag>/taudit-x86_64-linux.tar.gz.minisig
   minisign -V -p taudit.pub -m taudit-x86_64-linux.tar.gz
   ```

### Why this is "future work" and not shipped today

CI signing requires either (a) a passworded private key in a CI secret
(weak — anyone with `secrets:write` can exfiltrate it) or (b) a
maintainer-side post-release signing step. We will land (b) once a
real keypair is generated; the public key placeholder lives at
`release/taudit.pub` so the verification path is wired and ready to
swap in.

The placeholder file is **not** a valid signing key. Do not attempt to
verify against it. See `release/README.md` for status.
